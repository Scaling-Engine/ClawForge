# Phase 17.1: Context Hydration for Layer 1 - Context

**Gathered:** 2026-03-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a `get_project_state` tool to the LangGraph event handler (Layer 1) so the conversational agent can read `.planning/STATE.md` + `ROADMAP.md` from target repos via GitHub API. This gives the agent project awareness for better conversations ŌĆö it does NOT inject state into job descriptions (Layer 2 hydration is Phase 17.2).

</domain>

<decisions>
## Implementation Decisions

### Tool design
- Standalone `get_project_state` tool ŌĆö not auto-hydration inside `create_job`
- Instruction-driven: EVENT_HANDLER.md tells the agent when to call it ("Before creating a job for a target repo, call get_project_state to understand where the project is")
- Required `repo` parameter ŌĆö agent must specify which repo to check (slug/name/alias)
- Repo resolved via `loadAllowedRepos()` + `resolveTargetRepo()` ŌĆö same security boundary as `create_job`. Cannot read state from repos not in REPOS.json

### What to fetch
- Fetch two files: `.planning/STATE.md` and `.planning/ROADMAP.md`
- Truncation limits: STATE.md capped at ~4K chars, ROADMAP.md at ~6K chars, with `[TRUNCATED]` marker
- No git history fetch ŌĆö STATE.md already captures recent activity; Layer 2 can read git log from the clone
- When `.planning/` files don't exist: return structured response with nulls (`{ state: null, roadmap: null, message: "No GSD state found" }`)

### Injection point
- Tool output is for **conversational awareness only** ŌĆö the agent uses it to talk intelligently with the user ("you're on Phase 3, here are your blockers")
- Agent does NOT include project state in job descriptions for create_job ŌĆö Layer 2 context hydration is Phase 17.2's responsibility
- EVENT_HANDLER.md explicitly instructs the agent to call get_project_state before job creation for target repos

### Repo targeting
- Agent can check project state for **any allowed repo** in REPOS.json, not just the one it's about to target
- Works for clawforge itself (same-repo) ŌĆö useful for "what's the status of clawforge?"
- GitHub API for all lookups (including clawforge self-lookup) ŌĆö consistent code path, reads from main branch (latest merged state, not working tree)
- Fresh fetch every call ŌĆö no caching. Agent won't call this frequently; simplicity over optimization

### Claude's Discretion
- Exact truncation thresholds (4K/6K are guidelines, can adjust based on typical file sizes)
- Tool description wording
- How to structure the returned JSON (field names, formatting)
- EVENT_HANDLER.md instruction phrasing and placement

</decisions>
