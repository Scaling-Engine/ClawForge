## Spacing Scale

Declared values (multiples of 4 only). Source: existing codebase patterns — `chat-input.jsx` observed usage confirmed.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, remove-file badge offset (`-right-1.5` = 6px — closest to xs) |
| sm | 8px | Button padding (`p-2` = 8px), gap between toolbar icons |
| md | 16px | Default element spacing, textarea horizontal padding, form bottom padding |
| lg | 24px | Section padding |
| xl | 32px | Layout gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- Sub-mode dropdown: `px-1 py-1` (4px / 4px) — intentionally compact to sit flush beside the Code toggle button without adding visual weight
- Code toggle button and sub-mode `<select>`: `px-2 py-1` (8px / 4px) — matches existing terminal/code mode toggle sizing exactly
- Touch targets for all toolbar buttons: minimum 32px (current `p-2` on 16px icon = 32px clickable square) — acceptable for desktop-only tool; no mobile touch-target exception needed

## Color

Source: `templates/app/globals.css` — all values extracted from CSS custom properties.

| Role | Light Value | Dark Value | Usage |
|------|-------------|------------|-------|
| Dominant (60%) | `#ffffff` | `#0a0a0a` | Page background, surfaces |
| Secondary (30%) | `#f5f5f5` (`--muted`) | `#171717` | Chat input container (`bg-muted`), file preview strip, inactive toggle background |
| Accent (10%) | `#171717` (`--primary`) | `#ededed` | Active Code toggle button background (`bg-accent` maps to primary in practice — see note) |
| Destructive | `#dc2626` (light) / `#ef4444` (dark) | — | Error copy only — voice recording errors, streaming errors |

**Accent reserved for:** The active Code toggle button (`</>` when `codeActive === true`). No other element in Phase 48 uses the accent surface. The sub-mode `<select>` uses `text-muted-foreground` when inactive — it does NOT use accent.

**Note on `bg-accent`:** The codebase references `bg-accent text-accent-foreground` in existing toggle buttons (lines 280, 320 of `chat-input.jsx`) but `--accent` is not declared in `globals.css`. Tailwind v4 falls back to the `--primary` token for `bg-accent`. This is the existing project behavior — Phase 48 uses the same `bg-accent text-accent-foreground` pattern for the active Code toggle, inheriting whatever token resolution is in effect. Do not introduce a new `--accent` declaration.

**Inactive toggle state:** `text-muted-foreground hover:text-foreground` — matches all three existing toggles exactly.

**Disabled state:** No change to button color when `isStreaming` is true — opacity handled by cursor change only (existing pattern: `disabled={isStreaming}` without explicit disabled styling). Maintain this pattern.

**Sub-mode `<select>` when Code is active:** `text-xs bg-transparent border border-border rounded px-1 py-1 text-muted-foreground focus:outline-none` — does not use accent color. The select is subordinate to the toggle.

### Code Toggle Button (new unified element)

Replaces the three existing buttons: Terminal toggle (`>_`), Shell toggle (`$`), Code toggle (`</>`).

```jsx
// Unified Code toggle — always present (when admin), always routes to /stream/terminal
<button
  type="button"
  onClick={onToggleCode}
  className={cn(
    'inline-flex items-center justify-center rounded-lg px-2 py-1 text-xs font-mono',
    codeActive
      ? 'bg-accent text-accent-foreground'
      : 'text-muted-foreground hover:text-foreground'
  )}
  aria-label="Toggle Code mode"
  aria-pressed={codeActive}
  disabled={isStreaming}
  title="Code mode — routes to Claude Code CLI"
>
  {'</>'}
</button>
```

### Sub-Mode Select (new element, visible only when `codeActive`)

```jsx
{codeActive && (
  <select
    value={codeSubMode}
    onChange={(e) => onChangeCodeSubMode(e.target.value)}
    className="text-xs bg-transparent border border-border rounded px-1 py-1 text-muted-foreground focus:outline-none"
    disabled={isStreaming}
    aria-label="Code sub-mode"
  >
    <option value="plan">Plan</option>
    <option value="code">Code</option>
  </select>
)}
```

### Admin Guard

The Code toggle renders only when the user is admin. Non-admin users see no Code toggle — not a disabled toggle. The `session.user.role` prop flows from `chat-page.jsx` into `Chat` then `ChatInput`.

```jsx
// Only render the Code toggle for admin users
{isAdmin && onToggleCode && (
  // ... toggle button
)}
```

## Interaction States

| State | Code Toggle | Sub-mode Select | Textarea |
|-------|-------------|-----------------|----------|
| Default (codeActive: false) | Muted text, no background | Hidden | Normal font (sans) |
| Active (codeActive: true) | `bg-accent text-accent-foreground` | Visible, `plan` selected by default | Monospace font (`font-mono`) |
| Streaming | Same visual, `disabled={true}` | Same visual, `disabled={true}` | Disabled |
| Non-admin | Not rendered | Not rendered | Normal font |

**Mode persistence:** `codeActive` defaults to `false` on every new chat. No localStorage persistence. This is intentional — new chat = fresh intent.

**Textarea font switch:** When `codeActive` is true, textarea gains `font-mono` class. This behavior was already present for the old `codeMode` — carry it forward to the unified Code toggle.

## Accessibility Requirements

- Code toggle button: `aria-pressed` attribute reflects `codeActive` boolean — matches existing terminal toggle pattern
- Sub-mode `<select>`: native `<select>` element — keyboard accessible by default, screen readers announce option changes
- Disabled state on streaming: `disabled={isStreaming}` on both toggle and select — browsers handle `aria-disabled` automatically for native elements
- Admin gate: hiding the toggle (not disabling it) for non-admins prevents confusion — non-admins have no Code toggle to misinterpret
