**Researched:** 2026-03-06

## Summary

This phase adds a single `get_project_state` LangGraph tool to the event handler so the conversational agent can fetch `.planning/STATE.md` and `.planning/ROADMAP.md` from any allowed repo via the GitHub Contents API. The implementation is straightforward -- all building blocks exist: `githubApi()` for authenticated GitHub requests, `loadAllowedRepos()` + `resolveTargetRepo()` for repo resolution, and the established `tool()` + `z` pattern for tool definition.

The main work is: (1) a `fetchRepoFile()` helper in `lib/tools/github.js` that fetches a single file's content via `GET /repos/{owner}/{repo}/contents/{path}`, (2) a `getProjectStateTool` in `lib/ai/tools.js` that calls it for both files with truncation, (3) registering the tool in `lib/ai/agent.js`, and (4) updating `instances/noah/config/EVENT_HANDLER.md` with instructions for when to call it.

**Primary recommendation:** Follow the exact patterns established by `createJobTool` -- same imports, same `tool()` + `z.object()` shape, same `JSON.stringify()` return. Use `application/vnd.github.raw+json` Accept header for file fetches to avoid base64 decoding complexity.

### Pattern 2: GitHub File Fetch Helper
**What:** New `fetchRepoFile()` function in `lib/tools/github.js`
**Why separate:** Reusable for Phase 17.2 (Layer 2 hydration) and any future file-reading needs
**Example:**
```javascript
// Source: GitHub REST API docs (https://docs.github.com/en/rest/repos/contents)
async function fetchRepoFile(owner, repo, path, { ref = 'main', maxChars = null } = {}) {
  const endpoint = `/repos/${owner}/${repo}/contents/${path}`;
  const params = ref ? `?ref=${ref}` : '';

  const res = await fetch(`https://api.github.com${endpoint}${params}`, {
    headers: {
      'Authorization': `Bearer ${process.env.GH_TOKEN}`,
      'Accept': 'application/vnd.github.raw+json',
      'X-GitHub-Api-Version': '2022-11-28',
    },
  });

  if (res.status === 404) return null;
  if (!res.ok) throw new Error(`GitHub API error: ${res.status}`);

  let content = await res.text();
  if (maxChars && content.length > maxChars) {
    content = content.slice(0, maxChars) + '\n\n[TRUNCATED]';
  }
  return content;
}
```

**Key design choice:** Use `application/vnd.github.raw+json` Accept header instead of decoding base64. This returns raw file content directly as text, avoiding the base64 decode step entirely. The existing `githubApi()` helper hardcodes `application/vnd.github+json` so we need a direct fetch call (or a new helper) rather than calling `githubApi()`.

**What:** Import tool in `agent.js`, add to tools array
**Example:**

**CRITICAL:** Per STATE.md decision `[v1.3 roadmap]`: "Tool stub must be registered in agent tools array before any EVENT_HANDLER.md intake is written -- avoids SQLite checkpoint corruption on tool add mid-session." Both tool registration and EVENT_HANDLER.md update happen in the same deployment, so this is fine as long as both are in the same commit/PR.

### Anti-Patterns to Avoid
- **Do not reuse `githubApi()` for raw file fetches** -- it hardcodes the Accept header to `application/vnd.github+json` which returns base64-encoded content. Use a direct fetch with `application/vnd.github.raw+json` or create a dedicated `fetchRepoFile()`.
- **Do not cache results** -- per user decision, fresh fetch every call. Simplicity over optimization.
- **Do not inject state into job descriptions** -- that is Phase 17.2's responsibility. This tool is for conversational awareness only.

## Common Pitfalls

### Pitfall 1: SQLite Checkpoint Corruption on Tool Change
**What goes wrong:** Adding a tool to the agent's tools array while existing checkpointed conversations reference the old tool set can cause deserialization issues.
**Why it happens:** LangGraph's SQLite checkpointer stores the conversation state including tool call history. If the tool set changes, old checkpoints may reference tools that don't exist (or miss new ones).
**How to avoid:** Deploy tool registration + EVENT_HANDLER.md update together in one deployment. After deployment, existing conversations will see the new tool on their next message (the agent singleton is recreated on container restart).
**Warning signs:** Agent errors on resumed conversations after deployment.

### Pitfall 2: 404 vs Error Handling
**What goes wrong:** Treating a 404 (file not found) the same as a 500 (API error) causes the tool to report failures when repos simply don't have `.planning/` directories.
**Why it happens:** Not all repos use GSD. A missing STATE.md is expected, not an error.
**How to avoid:** Return `null` for 404s, throw only on actual API errors (5xx, auth failures). The tool should return a structured "no GSD state found" message, not an error.
**Warning signs:** Agent telling the user "failed to check project state" when the repo just doesn't have planning files.

### Pitfall 3: Large File Truncation Edge Cases
**What goes wrong:** Truncating mid-YAML-frontmatter or mid-markdown-table creates malformed output that confuses the LLM.
**Why it happens:** Naive character-count truncation doesn't respect document structure.
**How to avoid:** The `[TRUNCATED]` marker makes it clear the content is cut off. For the ~4K/6K limits, this is pragmatic -- STATE.md and ROADMAP.md are structured documents where the most important content (current position, next phases) is near the top. No need to parse structure for truncation.

### Pitfall 4: GH_TOKEN Scope
**What goes wrong:** The PAT used by the event handler may not have access to all repos in REPOS.json.
**Why it happens:** Fine-grained PATs are scoped to specific repos. If a new repo is added to REPOS.json but the PAT doesn't include it, file fetches will fail with 404 or 403.
**How to avoid:** The tool should handle 403 gracefully (return a message like "no access to this repo's planning files") rather than throwing. Document this in error handling.

## Code Examples

### Complete Tool Implementation Pattern
```javascript
// Source: Derived from existing tools.js patterns + GitHub API docs
const getProjectStateTool = tool(
  async ({ repo }) => {
    const repos = loadAllowedRepos();
    const resolved = resolveTargetRepo(repo, repos);
    if (!resolved) {
      return JSON.stringify({
        success: false,
        error: `Repo "${repo}" not recognized or not allowed. Available: ${repos.map(r => r.name).join(', ')}`,
      });
    }

    const { owner, slug } = resolved;
    const [state, roadmap] = await Promise.all([
      fetchRepoFile(owner, slug, '.planning/STATE.md', { maxChars: 4000 }),
      fetchRepoFile(owner, slug, '.planning/ROADMAP.md', { maxChars: 6000 }),
    ]);

    if (!state && !roadmap) {
      return JSON.stringify({
        success: true,
        repo: `${owner}/${slug}`,
        state: null,
        roadmap: null,
        message: 'No GSD planning state found in this repo.',
      });
    }

    return JSON.stringify({
      success: true,
      repo: `${owner}/${slug}`,
      state: state || null,
      roadmap: roadmap || null,
    });
  },
  {
    name: 'get_project_state',
    description:
      'Read the project planning state (.planning/STATE.md and ROADMAP.md) from a target repo. ' +
      'Use this to understand where a project stands before creating jobs or answering questions about project status. ' +
      'Returns current phase, progress, decisions, and roadmap.',
    schema: z.object({
      repo: z.string().describe(
        'Repository name or alias (e.g., "neurostory", "ns", "clawforge", "cf"). ' +
        'Must match an entry in the allowed repos list.'
      ),
    }),
  }
);
```

### fetchRepoFile Error Handling
```javascript
async function fetchRepoFile(owner, repo, filePath, { ref = 'main', maxChars = null } = {}) {
  const endpoint = `/repos/${owner}/${repo}/contents/${encodeURIComponent(filePath)}`;
  const params = ref ? `?ref=${ref}` : '';

  try {
    const res = await fetch(`https://api.github.com${endpoint}${params}`, {
      headers: {
        'Authorization': `Bearer ${process.env.GH_TOKEN}`,
        'Accept': 'application/vnd.github.raw+json',
        'X-GitHub-Api-Version': '2022-11-28',
      },
    });

    if (res.status === 404) return null;
    if (res.status === 403) return null; // No access -- treat same as not found
    if (!res.ok) {
      console.error(`GitHub API error fetching ${owner}/${repo}/${filePath}: ${res.status}`);
      return null;
    }

    let content = await res.text();
    if (maxChars && content.length > maxChars) {
      content = content.slice(0, maxChars) + '\n\n[TRUNCATED]';
    }
    return content;
  } catch (err) {
    console.error(`Failed to fetch ${owner}/${repo}/${filePath}:`, err.message);
    return null; // Non-fatal -- tool should still return partial results
  }
}
```

**Key:** All errors return `null` rather than throwing. The tool aggregates results and reports what it found. A complete failure (both files null) produces the "No GSD state found" message.

## Open Questions

1. **Path encoding for `.planning/STATE.md`**
   - What we know: GitHub Contents API paths use forward slashes. The `.planning/` prefix with a dot should work fine.
   - What's unclear: Whether `encodeURIComponent` is needed for the path (dots and slashes are valid in URL paths).
   - Recommendation: Use the path as-is without encoding the full thing -- only encode individual path segments if they contain special chars. The GitHub API handles dotfiles fine.

2. **REPOS.json location for `loadAllowedRepos()`**
   - What we know: `loadAllowedRepos()` reads from `config/REPOS.json` relative to PROJECT_ROOT. In Docker, each instance has its own config directory overlaid.
   - What's unclear: Nothing -- this is well-established.
   - Recommendation: Use as-is. The function already handles this correctly.
