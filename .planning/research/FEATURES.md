# Feature Research: v3.0 Customer Launch

**Domain:** Developer-tool SaaS — self-service onboarding, billing/access control, in-app documentation, multi-tenant health monitoring
**Researched:** 2026-03-17
**Confidence:** MEDIUM (WebSearch verified; no official docs for proprietary patterns; all findings attributed)
**Scope:** NEW features only for v3.0. Everything through v2.2 is shipped and not re-researched.

## Context: What Is Being Built

v3.0 adds commercial launch capabilities to the existing ClawForge platform. These are additive to the complete v2.2 system:

- **Self-service onboarding** for new instance operators (setup wizard, progress tracking, first-value moments)
- **Billing / access control** — per-customer instance limits, usage tracking, tiered access
- **In-app documentation and help** — tooltips, guided tours, operator documentation strategy
- **Team monitoring dashboard** — all instances health, active jobs, errors, cross-instance status

**What is confirmed shipped (v2.2 — do not rebuild):**

| Capability | Location |

## Area 1: Self-Service Operator Onboarding

### What This Is

Today, a new ClawForge instance operator follows a setup checklist in the PR generated by `createInstanceJob` — a static markdown document. The operator works through it manually: configuring Slack apps, GitHub PATs, Docker Compose, DNS. There is no progress tracking, no in-app guidance, and no signal to the platform that setup is complete.

Self-service onboarding adds three things: (1) a wizard UI that tracks which steps are done, (2) a "first-value moment" that confirms the setup works before the operator leaves the wizard, and (3) a persistent checklist the operator can return to.

**Industry standard (MEDIUM confidence — multiple SaaS sources agree):** Time-to-first-value is the core onboarding metric. For developer tools, first value = first successful job run, not account creation. Onboarding flows over 20 steps lose 30-50% of completions. The sweet spot is 3-7 core steps with progressive disclosure for advanced config.

### Table Stakes

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Onboarding checklist with progress state** | Without progress tracking, operators have no idea where they are in the setup sequence or what remains. Static markdown checklists don't persist state. Any serious setup flow shows a checklist with checked/unchecked state that survives page reloads. | LOW | DB-backed checklist: new `onboarding_steps` table keyed by `(instanceId, stepKey)`. Step keys: `docker_configured`, `slack_configured`, `github_pat_set`, `first_repo_added`, `first_job_run`. Server Action `markStepComplete(instanceId, stepKey)`. UI: checklist widget in the admin panel sidebar or a dedicated `/admin/setup` page. Reuses existing Drizzle ORM + SQLite patterns. |
| **Automated step verification** | "Connected" checkmarks that the operator manually clicks are unreliable — an operator clicks "done" before actually testing. Steps that can be verified programmatically must be. At minimum: GitHub PAT validity check, Docker socket reachability, Slack webhook test ping. | MEDIUM | Per-step verification functions: `verifyGitHubPat()` — call GitHub `/user` API with the configured PAT, check 200; `verifyDockerSocket()` — call `docker.info()` via dockerode, check response; `verifySlackWebhook()` — POST test message to configured webhook URL. Steps auto-check on page load and mark verified. Manual override available if auto-check fails for environment reasons. |
| **First-job success as the "first-value moment"** | Operators don't experience value by filling out a form — they experience value when the first job runs and produces a PR. The onboarding wizard should have a final step: "Send your first job" with a pre-filled example prompt. Completion of this step marks onboarding as complete. | MEDIUM | Final onboarding step: a simplified job-dispatch widget with a pre-filled prompt ("Add a README section describing this repo's purpose") targeting the configured test repo. On job completion (success or failure), onboarding marks the `first_job_run` step complete. UI transitions from wizard to "you're set up" confirmation. Reuses existing `createJob()` dispatch and `getJobStatus()` polling. |
| **Resumable setup across sessions** | Operators set up ClawForge over hours or days (waiting for Slack app approval, GitHub PAT permissions). The checklist state must persist so they can return and continue. | LOW | The `onboarding_steps` DB table provides this automatically. `/admin/setup` reads checklist state on load. Incomplete steps rendered as todo, complete steps rendered as checked. Operators can bookmark the setup page. No additional complexity beyond the DB table. |

### Differentiators

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Instance health scorecard after setup** | After first job runs, show a health summary: Docker status, GitHub PAT scope check, Slack connectivity, job dispatch latency. Turns onboarding completion into an operational baseline. | MEDIUM | Extends the verification functions from table stakes. New `/admin/health` page that runs all checks on demand and shows last-checked timestamps. This page also serves as the operator's "is everything working?" page post-onboarding. Useful beyond initial setup. |
| **Setup time estimate per step** | Show estimated time for each step ("~2 minutes: Create GitHub PAT"). Reduces abandonment by setting expectations. Userpilot data shows pre-disclosed time estimates increase onboarding completion rates. | LOW | Static labels on each step. No dynamic calculation needed. "~5 minutes: Configure Slack app" is a one-time copy addition. Extremely low complexity. |
| **Onboarding video or screenshot per step** | Visual confirmation of what each configuration screen looks like. Slack app creation has ~6 sub-steps that are confusing without screenshots. | MEDIUM | Static assets per step: screenshots or short GIF recordings embedded in the wizard. Content-heavy but code-light. Doubles as documentation. Risk: screenshots go stale when Slack/GitHub UIs change. Mitigate by linking to official docs instead of embedding screenshots. |

### Anti-Features

| Anti-Feature | Why Avoid | Alternative |
|--------------|-----------|-------------|
| **Full product tour on first login** | Linear walkthroughs that highlight every UI element are irrelevant to operators who haven't completed setup. They interrupt rather than guide. Industry data shows step-based tours triggered before setup completion are almost always dismissed immediately. | Trigger the guided tour (if built at all) only after `first_job_run` is confirmed. Pre-setup: show the setup wizard only. Post-setup: offer an optional "explore the platform" tour. |
| **Mandatory onboarding before any admin access** | Blocking admin panel access until onboarding is complete frustrates operators who know what they're doing and want to configure manually. | Show the onboarding wizard as a prominent banner/widget, not a blocking gate. Operators can dismiss it and proceed directly to the admin panel. The wizard is a helper, not a wall. |
| **Email-based onboarding drip sequences** | Sending setup reminder emails requires an email service (Resend, SendGrid), email templates, and unsubscribe flows. For a platform with 2-10 instances, this is massively over-engineered with no clear ROI. | In-app checklist is sufficient. If the operator has access to the admin panel, they don't need email reminders. Add emails only when operator count justifies it. |
| **AI-assisted onboarding wizard ("tell me about your use case")** | Natural language intake for setup is appealing but complex. The existing `createInstanceJob` conversation flow already captures use-case context for instance scaffolding. Duplicating this in a web-based wizard adds maintenance burden. | Keep AI-based instance creation in the existing chat flow. The web wizard handles configuration verification, not requirements gathering. |

## Area 2: Billing / Access Control

### What This Is

Access control for v3.0 means: limiting what each operator can do based on a plan tier or manual configuration, and tracking usage against those limits. This is not full SaaS billing infrastructure (no Stripe, no subscription management) — it's the platform-side entitlement layer that the platform controls.

**Industry standard (MEDIUM confidence — WebSearch verified):** Usage-based billing is now standard for AI/automation SaaS tools. However, billing infrastructure (Stripe Billing, Orb, Lago) solves invoicing, not enforcement. Enforcement of limits lives in application code. The pattern is: plan tier stored in DB → `checkEntitlement(instanceId, feature)` called before privileged operations → hard or soft limit enforced.

**ClawForge-specific context:** At v3.0 launch, ClawForge has 2 instances (Noah/Archie and Epic/StrategyES). The billing problem is not "charge credit cards" — it is "ensure instances can only consume their allocated resources and that superadmin can adjust those limits without code changes."

### Table Stakes

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Per-instance resource limits (jobs, workspaces, concurrent containers)** | Without limits, one misbehaving instance can exhaust Docker resources on the host, bringing down all instances. Limits are also the mechanism for tiered plans. | MEDIUM | New `instance_limits` table: columns `instanceId`, `maxConcurrentJobs`, `maxWorkspaces`, `maxJobsPerDay`, `planTier` (free/starter/pro). Read by `lib/tools/create-job.js` before dispatching: `checkJobLimit(instanceId)` — count active containers, compare against `maxConcurrentJobs`. If at limit, return error to operator. Same pattern for workspace creation in `lib/ws/manager.js`. Default limits baked in for existing instances. |
| **Usage tracking for jobs and token consumption** | Operators (and superadmin) need to see how many jobs were run this week, total tokens consumed, and error rate. Without this, there's no data to justify plan pricing or identify abuse. | MEDIUM | Extend `job_outcomes` table: add `promptTokens INT`, `completionTokens INT`, `durationSeconds INT` columns. These are populated by the job container's final summary step — the container already reads the JSONL output to find token counts. New `getUsageSummary(instanceId, since)` DB query. Superadmin portal usage page shows per-instance totals. |
| **Plan tier stored per instance (superadmin-editable)** | Superadmin must be able to adjust instance limits without deploying code. A `planTier` column on the instance config row (or `instance_limits` table) lets superadmin set `pro` → doubles job limit, enables clusters. | LOW | The `instance_limits` table described above includes `planTier`. Superadmin UI: a select dropdown in the instance management page to change the tier. When tier changes, limits update from a `PLAN_LIMITS` config object: `{ free: { maxConcurrentJobs: 2, ... }, starter: { ... }, pro: { ... } }`. No Stripe integration required. |
| **Graceful limit error messages** | When an operator hits a limit (e.g., 3 jobs already running), the error message must explain why the job was not dispatched and what to do. "Job limit reached" with no context is a bad UX. | LOW | `checkJobLimit()` returns a structured error: `{ allowed: false, reason: 'concurrent_jobs', current: 3, limit: 3, message: "3 jobs are currently running. Wait for one to complete or upgrade your plan." }`. This string is returned to the operator's channel (Slack/Telegram/chat) verbatim. No new infrastructure needed — reuses existing notification paths. |

### Differentiators

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Usage dashboard per instance** | A chart showing job volume over time, token consumption, and success/failure rates gives operators visibility into their own usage patterns. Useful for identifying runaway automation and planning capacity. | MEDIUM | Simple time-series query on `job_outcomes` grouped by day. Renders as a bar chart in the admin panel. Uses existing DB, no external analytics service. Use a lightweight charting library (recharts or chart.js — already potentially in use via superadmin portal). |
| **Soft limits with operator warnings** | Instead of hard-blocking at the limit, warn the operator when they reach 80% of their daily job budget. "You have used 8 of 10 daily jobs." This prevents surprise hard-stops. | LOW | Add warning threshold check (80%) to `checkJobLimit()`. If between 80% and 100%, dispatch succeeds but a warning notification is sent. If at 100%, block with the limit error. Threshold configurable per plan in `PLAN_LIMITS`. |
| **Stripe integration for future monetization** | When ClawForge has enough external customers to justify it, Stripe Billing handles subscription management, invoicing, and plan upgrades. | HIGH | Defer. Not needed for v3.0 launch with 2-10 known customers. The `planTier` column in `instance_limits` is the hook point for future Stripe integration: Stripe webhook updates `planTier` on subscription change. Build the entitlement layer now, wire Stripe later. |

### Anti-Features

| Anti-Feature | Why Avoid | Alternative |
|--------------|-----------|-------------|
| **Full billing infrastructure (Stripe, Orb, Lago) at v3.0 launch** | Implementing usage-based billing with Stripe requires: metering events, usage records API calls, subscription items, webhook handler, invoice portal, customer portal. For 2-10 instances, this is months of work that doesn't move the product forward. | Manual invoicing backed by usage data from the `job_outcomes` table. Superadmin exports a CSV of job counts. Stripe when volume justifies it. |
| **Token-level billing (charge per Claude token)** | Per-token billing requires real-time metering, overage detection, and customer communication about unexpected charges. It is the right end-state but terrible for early customer relationships where job quality is still being calibrated. | Plan-tier limits (jobs per day, concurrent containers). Simple, predictable, easy to communicate. Introduce token budgets when usage patterns are understood. |
| **Usage limits on the Event Handler (Layer 1 chat)** | Limiting how many chat messages an operator can send creates friction with no benefit — the LangGraph agent is cheap and fast. Limits that matter are job container dispatch (expensive) and workspace sessions (Docker resources). | Only apply resource limits to job dispatch and workspace creation. Chat is unlimited. |

## Area 3: In-App Documentation and Help

### What This Is

Operator documentation for v3.0 has two distinct surfaces: (1) static external docs (deployment runbook, config reference, troubleshooting guide) that operators read before and during setup, and (2) in-app contextual help (tooltips, empty states, inline explanations) that guide operators within the product.

**Industry pattern (MEDIUM confidence — multiple sources):** Developer tool SaaS needs both. External docs for setup and reference; in-product help for context-sensitive guidance. The critical distinction: in-product guided tours are effective post-activation (after first value); pre-activation tours are almost always dismissed. For a technical audience, good empty states and field-level tooltips outperform feature tours 3:1 in actual usage.

**ClawForge-specific constraint:** The platform has complex configuration surface (REPOS.json fields, MCP server config, AGENT_* secret conventions, quality gate arrays). New operators get lost in this without contextual help. The biggest documentation needs are: what AGENT_* prefix means, what fields REPOS.json accepts, what each quality gate command does, and what the entrypoint.sh configuration options are.

### Table Stakes

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Contextual tooltip on complex fields** | Admin panel fields like "AGENT_LLM_ prefix" or "mergePolicy" have non-obvious meanings. Without inline explanation, operators guess or abandon. Tooltips on field labels with one-sentence explanations are the minimum viable help system. | LOW | HTML `title` attribute or a small info icon (`ⓘ`) next to field labels that shows a tooltip on hover. No new library needed — pure CSS + HTML. Content: one sentence per field explaining what it does and what valid values are. Start with the 10 most-asked-about fields (AGENT_* prefix, mergePolicy, qualityGates, dispatchMethod, maxConcurrentJobs). |
| **Helpful empty states** | When an operator has no repos configured, the job dispatch UI shows a confusing empty state. When no MCP servers are configured, the MCP page is blank. Empty states should explain what the feature does and provide a direct action to add the first item. | LOW | Per-page empty state components: "No repos configured yet. Add your first repo to start dispatching jobs." + "Add Repo" button. Replaces silent empty lists. Small copy change, no infrastructure. Applies to: repos page, MCP servers page, GitHub secrets page, cluster runs list. |
| **External operator documentation (deployment runbook, config reference)** | Without external docs, every new operator emails for help. The existing CLAUDE.md and PROJECT.md are developer notes, not operator-facing docs. Operators need: how to deploy on a VPS, what env vars are required, how to configure REPOS.json, what the AGENT_* convention means, how to troubleshoot failed jobs. | MEDIUM | Static documentation site or markdown files in a `docs/` directory. Two documents minimum: (1) `DEPLOYMENT_RUNBOOK.md` — step-by-step VPS deploy, Docker Compose setup, DNS config, Slack app creation; (2) `CONFIG_REFERENCE.md` — every env var, every REPOS.json field, every admin panel setting, with defaults and valid values. These already partially exist in `docs/` — audit and fill gaps. No new infrastructure needed; GitHub renders markdown. |
| **Troubleshooting guide with common errors** | Operators will hit predictable failures: Docker socket permission denied, GitHub PAT scope missing, Slack signature verification failed, job container OOM. Each has a specific fix. A troubleshooting doc with the error message + fix reduces support burden. | MEDIUM | `TROUBLESHOOTING.md` in `docs/`. Format: error message (copy-pasteable) → cause → fix → prevention. Start with the 10 most common errors identified from v2.x development. Content-heavy, code-light. |

### Differentiators

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Post-first-job guided tour** | After an operator completes their first job, offer an optional tour: "Want to see what else ClawForge can do?" Tour highlights: cluster runs, workspaces, MCP servers. 3-5 steps maximum. Triggered only after `first_job_run` onboarding step is complete. | MEDIUM | Custom implementation: a `TourStep` component that highlights a target element, shows a tooltip with text, and provides prev/next buttons. Store tour state (completed/dismissed) in `localStorage` or the DB. Do not use a 3rd-party tour library (Shepherd.js, Intro.js) — they add JS bundle weight and vendor lock-in for what amounts to 3-5 annotated tooltips. |
| **Inline code examples in config fields** | REPOS.json and MCP_SERVERS.json fields benefit from showing a valid example value directly in the form. "e.g., `strategyes-lab/my-repo`" in the repo URL field prevents format mistakes. | LOW | `placeholder` attributes on form inputs. Static examples, no logic. Apply to all text fields in the repo CRUD form and MCP server form. Zero infrastructure. |
| **Video walkthrough for first deploy** | A 5-10 minute screen recording of a complete ClawForge deploy from scratch. Video is the highest-converting documentation format for "how do I set this up" questions. | LOW (content) / N/A (code) | Not a code change — a recorded video uploaded to YouTube or Loom. Linked from the onboarding wizard and the deployment runbook. Zero code complexity. High operator value for first-time setup. |

### Anti-Features

| Anti-Feature | Why Avoid | Alternative |
|--------------|-----------|-------------|
| **3rd-party in-app tour library (Shepherd.js, Intro.js, Appcues)** | Tour libraries add JS bundle weight, SaaS pricing ($X/MAU for hosted solutions), and DOM manipulation that breaks with Next.js App Router. For 3-5 tour steps, a 3rd-party library is over-engineered. | Custom tour component: a positioned tooltip div + a step counter. 50 lines of React. No vendor dependency. |
| **AI-powered help assistant in the admin panel** | An AI chat assistant that answers questions about ClawForge configuration is appealing but creates a support surface with unpredictable answers. The platform already has an AI chat — adding a second AI chat for documentation questions causes confusion about what each chat is for. | Good static docs + contextual tooltips answer 90% of questions without hallucination risk. Add AI help search only when static docs are proven insufficient. |
| **Full interactive product demo from the marketing site** | A Arcade/Storylane-style interactive walkthrough embedded on a marketing site is relevant for consumer SaaS, not for a developer tool sold to technical operators who evaluate by reading docs and running the product. | A short video walkthrough (see differentiators) + GitHub repo with clear README is the right demo surface for this audience. |

## Area 4: Team Monitoring Dashboard

### What This Is

The v2.2 superadmin portal has a health dashboard with per-instance cards showing active job count, runner status, and last job timestamp (30-second auto-refresh). v3.0 extends this into a proper operations dashboard with error tracking, alerting, and historical views.

**Context:** ClawForge already has the data model and API proxy pattern needed for multi-instance monitoring. The `queryAllInstances()` function in the superadmin API uses `Promise.allSettled()` for graceful partial results. The v3.0 monitoring work is primarily: (1) surfacing more data in the existing dashboard, (2) adding error tracking that persists across restarts, and (3) adding alert notifications when instances go unhealthy.

**Industry pattern (MEDIUM confidence — New Relic, AWS SaaS Lens):** Effective multi-tenant monitoring distinguishes systemic issues (affects all instances) from localized issues (one instance). The key metrics per instance: job success rate (last 24h), P95 job duration, active containers vs. limit, error rate by type. Tenant-aware filtering (show only errors from StrategyES) is the primary usability requirement.

### Table Stakes

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **Error tracking with persistence across restarts** | Currently, errors in job containers and the event handler appear in Docker logs which are lost on container restart. Operators investigating post-mortem failures have no error record. An error log that persists in SQLite is the minimum for any production system. | MEDIUM | New `error_events` table: columns `id`, `instanceId`, `level` (error/warn), `source` (job_container/event_handler/workspace), `message TEXT`, `jobId` (nullable FK to job_outcomes), `createdAt`. Populated by: (a) job container failures captured in `waitAndNotify()` when job exits with non-zero code; (b) event handler unhandled exceptions via a top-level error handler; (c) workspace crash events (already fired as notifications — persist to DB as well). New admin page `/admin/errors` shows recent errors with filters. |
| **Job success rate per instance** | A simple metric: what % of jobs in the last 24h succeeded. Without this, operators can't tell if their instance is healthy or degraded. | LOW | Query against existing `job_outcomes` table: `COUNT(*) WHERE instanceId = X AND createdAt > 24h ago` partitioned by `status` (merged/failed/cancelled). Percentage = merged / (merged + failed). Display on superadmin health card and per-instance admin panel. No new table needed. |
| **Alert on instance health degradation** | When an instance's job success rate drops below a threshold (e.g., 3 consecutive failures), or the runner goes offline, superadmin should receive a Slack notification. Without alerting, issues are discovered by operators who notice something is wrong, not proactively. | MEDIUM | Alert check runs in the existing health poll (30s auto-refresh cycle in superadmin). If `consecutiveFailures > ALERT_THRESHOLD` (configurable, default 3), call `notifySlack()` with the instance name and failure summary. Track `consecutiveFailures` in the `instance_limits` table (add column). Alert fires at most once per hour per instance (deduplicated by last-alert timestamp). Reuses existing Slack notification infrastructure. |
| **Historical job timeline per instance** | The superadmin dashboard shows current state. Operators investigating "what happened yesterday afternoon" need a time-series view of job activity — when jobs ran, how long they took, which failed. | MEDIUM | Time-series query on `job_outcomes` grouped by hour: `SELECT date_trunc('hour', createdAt), COUNT(*), status FROM job_outcomes WHERE instanceId = X GROUP BY 1, 3`. Renders as a stacked bar chart (success/failed) in the superadmin portal instance detail page. Same lightweight charting library as usage dashboard (recharts). |

### Differentiators

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Container resource utilization tracking** | Job containers that consume excessive CPU or memory indicate runaway Claude Code sessions. Tracking peak CPU/memory per job (via `container.stats()` snapshot at completion) enables capacity planning and identifies resource leaks. | MEDIUM | Poll `container.stats({stream: false})` once before container removal in `waitAndNotify()`. Store peak CPU% and memory bytes in `job_outcomes` as new columns. Surface as P95 memory in the usage dashboard. `dockerode` `container.stats()` is already available — this is new read + DB write logic only. |
| **Cross-instance job failure pattern detection** | If the same error appears across multiple instances (e.g., "GitHub rate limit hit" or "Claude API 529"), it is a systemic issue. Grouping errors by message prefix in the `error_events` table and highlighting when the same error appears on 2+ instances adds diagnostic value. | LOW | Query: `SELECT message, COUNT(DISTINCT instanceId) FROM error_events WHERE createdAt > 1h GROUP BY message HAVING COUNT(DISTINCT instanceId) > 1`. Surface as a "systemic issues" section on the superadmin health page. Pure SQL aggregation — no new tables. |
| **Workspace session duration and idle timeout telemetry** | Tracking how long workspace sessions run and when they idle-stop gives operators insight into interactive usage patterns. Unusually long sessions may indicate a stuck terminal or forgotten workspace. | LOW | `code_workspaces` table already has `createdAt` and `updatedAt`. Add `terminatedAt` column populated on stop/destroy. Session duration = `terminatedAt - createdAt`. Surface as average session duration in the usage dashboard. Idle-stop events already fire as notifications — persist them to `error_events` with `level=warn`. |

### Anti-Features

| Anti-Feature | Why Avoid | Alternative |
|--------------|-----------|-------------|
| **OpenTelemetry / distributed tracing** | Full distributed tracing (traces spanning event handler → Docker container → GitHub → Slack) requires an OpenTelemetry collector, trace storage (Jaeger, Tempo), and instrumentation at every call site. For 2 instances and 10-50 jobs/day, this is massive infrastructure overhead with no operational benefit over structured logs. PROJECT.md explicitly lists OpenTelemetry as out of scope. | Structured error logging to SQLite (`error_events` table) + Docker stdout logging. Sufficient for the current scale. |
| **External monitoring services (Datadog, New Relic, Grafana Cloud)** | SaaS monitoring tools add per-seat or per-host costs, require agent installation, and create data exfiltration risk (job logs contain code, secrets references). For a platform that already stores its data in SQLite, exporting to an external service is unnecessary. | In-platform monitoring pages in the superadmin portal. All data stays in the existing SQLite DB. |
| **Real-time streaming event feed across all instances** | A live "activity feed" showing every job start/stop/tool-call across all instances would require a pub/sub layer (Redis pub/sub or WebSocket broadcast) that doesn't exist. The complexity-to-value ratio is poor. | Polling-based dashboard (existing 30s refresh). Per-instance drill-down for live streams. Cross-instance view is aggregate stats only. |
| **Pagerduty / OpsGenie integration for alerting** | Incident management integrations make sense at 100+ instances. For 2-10 instances, a Slack message to superadmin is the right alert channel. Pagerduty adds account management and on-call scheduling overhead for a platform that doesn't have SLAs yet. | Slack notification to a designated `#clawforge-alerts` channel. Already has all the infrastructure. Pagerduty if/when SLAs are introduced. |

```
├── consecutiveFailures column ──extends──> instance_limits table (same migration as Area 2)
├── Historical timeline chart ──queries──> existing job_outcomes table
└── Container stats capture ──extends──> waitAndNotify() before container removal

[Cross-Area Dependencies]
├── Area 2 instance_limits table ──shared-by──> Area 4 alert tracking (consecutiveFailures column)
├── Area 1 first_job_run step ──triggers──> Area 3 post-first-job tour
├── Area 1 setup verification ──requires──> Area 2 instance_limits exist (checks limits are configured)
└── All areas ──require──> existing instanceId propagation in DB tables (confirmed in v2.2 superadmin work)
```

### Dependency Notes

- **Area 2 and Area 4 share a DB migration**: `instance_limits` table serves both billing limits (Area 2) and alert state tracking (Area 4). Build both tables in a single migration to avoid two schema changes.
- **Area 3 has no blocking dependencies**: Tooltips, empty states, and external docs can be built in any order and in parallel with other areas.
- **Area 1 depends on Area 2 conceptually**: The first-job widget should respect job limits. Build Area 2 `checkJobLimit()` before the Area 1 first-job dispatch widget to avoid an onboarding flow that bypasses limits.
- **Area 4 error tracking is independent**: `error_events` table and capture logic can be built before the monitoring UI is built. Capturing errors in the table immediately improves observability even before the admin page exists.

## MVP Definition

### Launch With (v3.0 initial)

Minimum set to call the platform production-ready for external customers.

- [ ] **Onboarding checklist with progress persistence** — without it, operators are lost in setup
- [ ] **Automated step verification (GitHub PAT, Docker socket)** — prevents "it worked in setup but broke in prod" failures
- [ ] **Per-instance job and workspace limits** — prevents one instance from starving Docker resources
- [ ] **Usage tracking on job_outcomes (tokens, duration)** — data needed for every future billing decision
- [ ] **Graceful limit error messages** — operators must understand why a job was rejected
- [ ] **Contextual tooltips on complex admin fields** — reduces support burden for config questions
- [ ] **Helpful empty states on key pages** — repos, MCP, secrets pages currently show nothing
- [ ] **External docs: deployment runbook + config reference** — minimum bar for any production product
- [ ] **Error events table with persistence** — post-mortem debugging requires error history
- [ ] **Job success rate metric on superadmin health cards** — single most useful operational metric

### Add After Validation (v3.0.x)

Features to add once core is working and first external customers are onboarded.

- [ ] **Alert on consecutive job failures** — trigger when monitoring data confirms it's needed
- [ ] **Historical job timeline chart** — useful once there's enough job history to visualize
- [ ] **Post-first-job guided tour** — add after first operators complete onboarding and give feedback
- [ ] **Soft limits with warnings (80% threshold)** — add when operators hit hard limits unexpectedly
- [ ] **Instance health scorecard page** — extends onboarding verification into ongoing operations
- [ ] **External docs: troubleshooting guide** — write from real issues as they surface, not speculatively
- [ ] **Container resource utilization tracking** — add when job failures show resource exhaustion

### Future Consideration (v3.1+)

Features to defer until product-market fit is established.

- [ ] **Stripe integration for monetization** — when invoice volume justifies automation
- [ ] **Video walkthrough for first deploy** — production-quality content, not needed for first customers
- [ ] **Cross-instance failure pattern detection** — useful at 5+ instances, overkill at 2-3
- [ ] **Workspace session duration telemetry** — low signal value until interactive usage patterns emerge
